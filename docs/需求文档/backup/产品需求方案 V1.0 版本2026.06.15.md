# 产品需求方案 V1\.1 版本

> 版本变更说明（V1.1 基于 V1.0 客户反馈补充，2026.06.19）
>
> 1. Combo Card 审核升级为"套餐级审核"，以 `combo_id` 为审核主对象，同时输出套餐整体结论与单菜调整建议。
> 2. 规则卡 / 案例卡 / 红线卡字段补全为结构化判断资产，FastGPT 检索结果只引用不可替代。
> 3. 证据链增加"正式交付 Gate"，明确 preview / formal / blocked 的触发条件，报告导出区分 preview 与 formal。
> 4. H5 执行回传链接与证据上传补充最小安全边界（token 单卡绑定、撤销/过期、访问日志、提交版本保留、文件校验、路径隐藏）。
> 5. LLM 判断边界收紧：输出必须含 evidence_ids / rule_refs / confidence / missing_data；规则卡与红线卡优先级高于 LLM；失败/超时/超预算回退规则兜底或 dry-run。
>
> 说明：OCR / POS 接入里程碑暂不调整，按现有接口文档和接入条件继续推进。

# 产品定位

1. 为精品正餐品牌提供菜单决策闭环系统。从数据录入到执行回写，让每一次换菜、下架、改规格都有依据、有跟踪、有反馈。

2. 核心价值：把凭经验调菜单变成有证据、有规则、有闭环的菜单决策。

# 闭环主链路

```Plaintext
基础数据录入 → 门店定位与客群基线 → 菜品多维诊断 → 菜单健康度 → Combo Card 套餐级审核 → 作战卡生成与分配 → 门店执行回传
→ 执行后数据复核 → 老板反馈 → 规则/案例/红线回写
```

# 功能详设

## 基础数据录入

### CSV 导入

**页面**：菜品数据 → CSV 导入

**交互流程**：

1. 运营人员选择门店

2. 选择导入类型：菜品列表 / 成本数据 / 销量数据

3. 下载对应 CSV 模板

4. 上传填写好的 CSV 文件

5. 系统解析并展示预览（前 10 行）

6. 运营确认 → 写入数据库

7. 展示导入结果：成功 N 条、失败 N 条、失败原因列表

**CSV 模板定义**：

菜品列表：

```Plaintext
菜名,分类,售价
宫保鸡丁,热菜,38
```

成本数据：

```Plaintext
菜名,食材成本
宫保鸡丁,12.5
```

销量数据：

```Plaintext
菜名,统计周期起始,统计周期结束,销量
宫保鸡丁,2025-01-01,2025-01-31,156
```

**业务规则**：

- 菜名作为匹配键，成本/销量导入时按菜名关联已有菜品

- 菜名未匹配到的行标记为"未关联"，需运营手动关联或新建

- 同一菜品同一周期销量重复导入，覆盖旧值

- 售价和成本必须 \> 0，否则该行标记失败

- 单次导入上限 500 行

**API**: 

```Plaintext
POST /api/v1/stores/{store_id}/dishes/import
Content-Type: multipart/form-data

请求：
- file: CSV 文件
- import_type: "dishes" | "costs" | "sales"

响应：
{
  "data": {
    "total": 50,
    "success": 48,
    "failed": 2,
    "failures": [
      {"row": 12, "reason": "售价不能为0"},
      {"row": 35, "reason": "菜名未找到匹配"}
    ]
  }
}
```

#### 套餐管理

**页面**：菜品数据 → 套餐管理

**套餐表单字段**：

|**字段**|**类型**|**必填**|**说明**|
|---|---|---|---|
|套餐名称|文本|是|1\-50 字|
|包含菜品|多选关联|是|从本门店菜品中选择|
|套餐售价|数字|是|\> 0|
|套餐食材成本|数字|否|可自动汇总菜品成本，也可手动覆盖|
|平台团购标记|布尔|否|是否为平台团购套餐，影响定价与毛利判断|
|适用时段|单选|否|午市 / 晚市 / 全天|
|目标客群|多选标签|否|套餐面向客群标签，供套餐级审核使用|
|老板目标|下拉选择|否|引流 / 提毛利 / 招牌展示 / 清库存，供套餐级审核判断目标一致性|
|是否包含招牌菜|自动|\-|根据包含菜品与招牌保护清单自动计算|

**业务规则**：

- 套餐成本 = 各菜品食材成本之和（默认），可手动覆盖

- 套餐毛利 = （套餐售价 \- 套餐成本） / 套餐售价 × 100%

- 平台团购标记、适用时段、目标客群、老板目标用于 Combo Card 套餐级审核的整体判断

#### OCR 菜单解析

**页面**：菜品数据 → OCR 解析

**交互流程**：

1. 上传菜单图片（jpg/png）或 PDF

2. 系统调用 Qwen OCR，返回结构化草稿列表

3. 运营逐条校正：修改菜名、分类、售价

4. 勾选要导入的行 → 确认导入

5. 导入的菜品进入菜品列表，状态标记为"OCR 导入待确认"

**业务规则**：

- OCR 结果为草稿，不可直接成为正式数据

- 运营必须逐条确认或修正后才能导入

- 单次上传上限 10 页/10 张图

**API**:

```Plaintext
Content-Type: multipart/form-data

请求：
- file: 图片或PDF
- store_id: 门店ID

响应：
{
  "data": {
    "task_id": "ocr_20250117_001",
    "items": [
      {"row": 1, "name": "宫保鸡丁", "category": "热菜", "price": 38.0, "confidence": 0.95},
      {"row": 2, "name": "??鸡", "category": "热菜", "price": 42.0, "confidence": 0.6}
    ]
  }
}

POST /api/v1/ocr/parse/{task_id}/confirm

请求：
{
  "items": [
    {"row": 1, "name": "宫保鸡丁", "category": "热菜", "price": 38.0, "selected": true},
    {"row": 2, "name": "白切鸡", "category": "凉菜", "price": 42.0, "selected": true}
```

### 菜品多维诊断

**触发条件**：门店数据完整度检查 `can_diagnose = true`

**页面**：诊断中心 → 菜品多维诊断

**交互流程**：

1. 运营选择门店，点击"开始诊断"

2. 系统逐维度计算（同步或异步，预计 10\-30 秒）

3. 诊断完成后跳转到结果页

4. 结果页展示：菜品列表 \+ 7 维标签 \+ 可排序/筛选

#### 各维度计算规则

**毛利维度**

```Plaintext
单品毛利率 = (售价 - 食材成本) / 售价 × 100%

标签规则：
- 毛利率 >= 60%  → "高毛利"
- 毛利率 40%-60% → "中等毛利"
- 毛利率 < 40%   → "低毛利"
- 毛利率 < 品牌红线值 → 额外标记"低于红线"
- 成本数据缺失   → "数据缺失"
```

**销量/偏好维度**

```Plaintext
按同分类内销量排名：
- 前 20%        → "高销量"
- 中间 60%      → "中等销量"
- 后 20%        → "低销量"

销量趋势（需要2个以上周期数据）：
- 环比增长 > 10%  → "上升"
- 环比变化 ±10%   → "平稳"
- 环比下降 > 10%  → "下降"
- 单周期数据       → "趋势待定"
```

**品牌角色维度**

```Plaintext
优先使用人工标注值。

未标注时自动推断：
- 招牌菜保护清单中的 → "招牌菜"
- 销量前20% + 毛利前30% → "流量菜"
- 销量后30% + 毛利前20% → "利润菜"
- 销量后30% + 毛利后30% → "陪衬菜"
- 其余 → "常规菜"
```

**客群匹配维度**

```Plaintext
调用 LLM，输入：菜品信息 + 客群基线 + 品牌定位
输出：匹配/不匹配/待确认
依据：LLM 需返回判断理由（1-2句）

规则：
- 招牌菜默认标记"匹配"，不做不匹配判定
- LLM 返回不确定时标记"待确认"
- 客群基线未完成时标记"基线缺失"
```

> LLM 判断边界收紧（详见"LLM 判断边界"章节）：LLM 输出必须包含 `evidence_ids`、`rule_refs`、`confidence`、`missing_data`；没有证据或规则引用时只能输出"待确认"，不得给 high confidence。

**场景匹配维度**

```Plaintext
调用 LLM，输入：菜品信息 + 主力消费场景 + 时段
输出：匹配/不匹配/待确认
依据：LLM 需返回判断理由

规则同上
```

> 同样受 LLM 判断边界收紧规则约束。

**出品落地维度**

```Plaintext
优先使用人工标注值。

未标注时自动推断（调用 LLM）：
- 输入：菜品名称、分类、厨房已知信息
- 输出：易/中/难 + 理由
```

> 同样受 LLM 判断边界收紧规则约束，无证据/规则引用时不得给 high confidence。

**组合贡献维度**

```Plaintext
仅对出现在套餐中的菜品计算：
- 该菜品出现在 >= 2 个套餐中 → "核心搭配"
- 该菜品出现在 1 个套餐中   → "可替换"
- 该菜品未出现在任何套餐中 → "无搭配"
```

#### 健康度计算规则

**总分计算**：

```Plaintext
总分 = 结构得分 × 0.25 + 价格带得分 × 0.20 + 毛利得分 × 0.30 + 招牌保护得分 × 0.25
```

**各维度得分**：

|**维度**|**计算规则**|
|---|---|
|结构得分|分类齐全度 30% \+ 各分类菜品数均衡度 40% \+ 流量菜/利润菜/招牌菜配比合理度 30%。满分 100|
|价格带得分|价格带覆盖度（低/中/高各段有菜品）50% \+ 价格梯度合理性 50%。满分 100|
|毛利得分|整体毛利率达标情况 50% \+ 低毛利菜品占比 30% \+ 高毛利菜品占比 20%。满分 100。毛利率 \< 品牌红线直接扣 30 分|
|招牌保护得分|招牌菜是否有保护（不被建议下架）40% \+ 招牌菜毛利率是否健康 30% \+ 招牌菜销量是否在前 30%30%。满分 100|

**风险规则**：

|**风险类型**|**触发条件**|**风险等级**|
|---|---|---|
|毛利风险|菜单整体毛利率 \< 品牌红线|高|
|毛利风险|低毛利菜品占比 \> 30%|中|
|结构风险|某分类菜品数 \< 3|中|
|结构风险|流量菜占比 \< 15%|低|
|品牌风险|招牌菜销量排名在后 30%|高|
|品牌风险|招牌菜毛利率 \< 40%|中|
|价格带风险|高/中/低任一价格段菜品数为 0|中|

**修复路径**：每条风险自动关联修复建议，修复建议从规则库中检索匹配。

### Combo Card 套餐级审核

**页面**：诊断中心 → Combo Card 审核

**前置条件**：菜品多维诊断 \+ 菜单健康度已完成，且门店至少存在 1 个套餐（combo）

**审核主对象**：以 `combo_id`（套餐对象）为审核主对象，不再以单个 `dish_id` 作为唯一审核对象。一个套餐下可关联多个菜品。

**交互流程**：

1. 运营点击"生成审核结论"

2. 系统基于诊断结果 \+ 健康度风险 \+ 规则库，按套餐生成审核结论（套餐整体结论 \+ 单菜明细建议）

3. 审核结论按套餐分组展示，每个套餐卡片包含"套餐整体结论"和"单菜调整建议"两块

4. 运营可对套餐整体结论确认，也可对单菜明细逐条确认或标记需老板拍板

5. 确认后可生成作战卡（作战卡可挂在套餐整体或单菜维度）

#### 套餐整体字段

审核主对象需携带以下套餐整体字段：

|**字段**|**说明**|
|---|---|
|combo_id|套餐 ID，审核主对象|
|套餐名称||
|目标套餐价|套餐售价|
|套餐成本|各菜品食材成本之和，可手动覆盖|
|套餐毛利率|（套餐售价 \- 套餐成本） / 套餐售价 × 100%|
|平台团购标记|是否为平台团购套餐（影响定价与毛利判断）|
|午市 / 晚市|套餐适用时段|
|目标客群|套餐面向客群标签|
|老板目标|老板对该套餐的明确目标（如引流 / 提毛利 / 招牌展示）|
|是否包含招牌菜|套餐内是否含招牌保护菜|

#### 审核结论生成规则

**触发审核的套餐**（满足任一条件即进入审核）：

- 套餐毛利率低于品牌红线

- 套餐内任一菜品的客群匹配/场景匹配为"不匹配"

- 套餐内任一菜品出品难度为"难"且销量为"低销量"

- 套餐或其菜品诊断维度中 \>= 3 个维度标签为负面

- 套餐内含招牌菜但套餐毛利率低于红线

- 老板目标（如引流）与套餐实际表现不匹配

**套餐级风险类型**：

|**风险类型**|**说明**|
|---|---|
|毛利风险|套餐整体毛利率低于红线，或与目标毛利差距过大|
|品牌 / 客群风险|套餐定位与目标客群/品牌红线冲突，如商务宴请套餐含大量低成本小吃|
|出品落地风险|套餐内多道"难"出品菜品，厨房高峰期无法落地|
|组合价值风险|套餐组合价值感弱（如总价比单点仅低 5%），缺乏吸引力|

**单菜明细风险类型**：

- 建议替换：哪些菜建议替换（含替换候选及依据）

- 建议改规格：哪些菜建议改规格（如缩份、提价）

- 必须老板拍板：哪些菜（如下架招牌菜）必须老板拍板，系统不做自动决策

**审核输出结构**（每个套餐一条，同时保留套餐整体结论与单菜调整建议）：

- 多条建议互斥（如替换 A 菜 vs 保留 A 菜改规格）→ 标记拍板

- 证据缺口数量 \> 维度数量的 50% → 标记拍板

- 套餐整体结论与老板目标冲突（如老板目标引流但建议下架引流菜）→ 标记拍板

### 作战卡生成与分配

**页面**：作战卡 → 作战卡列表

**前置条件**：Combo Card 套餐级审核结论已生成（且关联审核结论为 formal），运营已确认至少 1 条

**交互流程**：

1. 运营在审核结论页勾选要生成作战卡的条目（套餐整体结论或单菜明细均可）

2. 点击"生成作战卡"

3. 系统为每条生成一张作战卡

4. 运营编辑作战卡内容（可修改动作、预期效果等）

5. 选择目标门店和执行人

6. 点击"分配"

7. 企微通知发送到店长

> 正式交付 Gate：作战卡关联审核结论必须为 formal（非 preview/blocked），否则禁止分配。

#### 作战卡字段

|**字段**|**类型**|**必填**|**说明**|
|---|---|---|---|
|作战卡 ID|自动生成|\-|格式：BC\-\{门店编号\}\-\{日期\}\-\{序号\}|
|关联审核结论|关联|是|关联 combo\_card\_audit（套餐整体或单菜明细）|
|关联套餐|关联|是|关联 1 个套餐|
|关联菜品|关联|否|挂在单菜级时关联 1 个菜品，套餐整体级可不填|
|动作类型|单选|是|换菜/下架/改规格/新增/调价|
|动作描述|文本|是|具体操作说明，如"将麻婆豆腐替换为蟹粉豆腐"|
|预期效果|文本|是|如"提升该位置毛利率 28%→55%"|
|证据需求|多条文本|否|执行后需要采集的证据，如"新菜单照片"、"替换后 1 周销量"|
|观察周期|数字\+单位|是|如 14 天 / 30 天|
|规则/案例来源|关联|否|关联规则卡 ID/案例卡 ID|
|目标门店|关联|是||
|执行人|文本|否|店长姓名|
|状态|自动|\-|待分配/待执行/执行中/已完成/无法执行|

#### 作战卡状态机

```Plaintext
生成 → 待分配 ──(运营分配)──→ 待执行 ──(店长确认接收)──→ 执行中
                                                                    │
                                              ┌─────────────────────┤
                                              ↓                     ↓
                                           已完成               无法执行
                                          (店长回传)           (店长回传)
```

状态流转规则：

- 待分配 → 待执行：运营分配门店后自动流转

- 待执行 → 执行中：店长首次打开作战卡时自动流转

- 执行中 → 已完成：店长提交执行回传

- 执行中 → 无法执行：店长选择"无法执行"并填写原因

- 已完成/无法执行不可回退

**关闭 Gate**：作战卡关闭前必须检查：执行回传已提交、执行证据已上传、执行后指标或缺数据说明已填写，缺一则不可关闭，系统显示 blocked 原因（记录于 battle\_card.close\_blocked\_reasons）。

**API**: 

```Plaintext
POST /api/v1/battle-cards/generate              # 从审核结论生成作战卡
GET  /api/v1/battle-cards                       # 作战卡列表（支持按门店/状态筛选）
GET  /api/v1/battle-cards/{id}                  # 作战卡详情
PUT  /api/v1/battle-cards/{id}                  # 编辑作战卡
POST /api/v1/battle-cards/{id}/assign           # 分配作战卡
GET  /api/v1/battle-cards/{id}/status-history   # 状态变更记录
```

### 门店执行回传

**页面**：H5 门店端

**访问方式**：

- 企微通知中包含链接

- 或运营在后台生成临时链接发给店长

- 链接格式：`/execution/{token}`，token 关联具体作战卡，有效期 7 天

**token 安全规则**：

- token 只能绑定单张作战卡，不能访问其他门店或其他作战卡（越权访问返回 403）

- token 必须支持过期、撤销和重新生成；运营可在后台主动撤销并重新生成

- 过期 / 已撤销 token 只能查看提示（"链接已失效，请联系运营重新生成"），不能继续提交或上传

- token 重新生成后，旧 token 立即失效

**访问日志**：每次打开链接、提交回传、上传证据都要记录访问日志（访问时间、token、动作类型、IP、User-Agent、成功/失败）。

**交互流程**：

1. 店长打开链接，看到作战卡内容

2. 选择执行状态：已执行 / 未执行 / 无法执行

3. 填写执行说明

4. 上传证据

5. 录入执行后指标（可选）

6. 提交

**回传表单字段**：

|**字段**|**类型**|**必填**|**条件**|
|---|---|---|---|
|执行状态|单选|是|已执行/未执行/无法执行|
|执行人|文本|是||
|执行日期|日期|是||
|实际动作|文本|选了"已执行"时必填|描述实际做了什么|
|未执行原因|文本|选了"未执行"或"无法执行"时必填||
|证据图片|图片上传|选了"已执行"时必填|支持 jpg/png，单张 \< 5MB，最多 5 张|
|证据备注|文本|否|补充说明|
|执行后指标\-观察周期|数字\+单位|否|如 7 天/14 天|
|执行后指标\-基线值|数字|否|执行前数值|
|执行后指标\-结果值|数字|否|执行后数值|

**文件上传安全规则**：

- 限制文件类型：仅允许 jpg / png（白名单）

- 限制单文件大小：\< 5MB

- 限制单次数量：最多 5 张

- 校验真实文件类型：通过文件头（magic bytes）校验，不可仅凭扩展名放行

- 上传文件路径不能直接暴露本地绝对路径，对外只返回经过签名/映射的访问 URL

**业务规则**：

- 提交人和提交时间由系统自动记录

- 同一作战卡可多次提交回传（补充证据），每次提交保留版本快照，不要只覆盖最后一次；执行状态以最新一次为准，但历史版本可追溯

- 图片存储到本地文件系统（或对象存储），数据库只记录映射后的访问 URL，不记录本地绝对路径

**API**: 

```Plaintext
GET  /api/v1/execution/{token}                  # 获取作战卡详情（校验 token 有效性、越权、过期）
POST /api/v1/execution/{token}/submit           # 提交执行回传（保留版本快照）
POST /api/v1/execution/{token}/evidence         # 补充上传证据（校验文件类型/大小/数量/真实类型）
POST /api/v1/battle-cards/{id}/token/revoke     # 撤销 token（运营）
POST /api/v1/battle-cards/{id}/token/regenerate # 重新生成 token（运营）
GET  /api/v1/battle-cards/{id}/access-logs      # 查看访问日志
```

#### 正式交付 Gate

当前方案已有 `preview / formal / blocked`，V1.1 明确什么情况下不能正式交付。证据链作为正式交付的硬性 Gate，缺口未补齐时系统必须明确显示 blocked 原因。

**只能生成 `preview`、不能生成正式报告的证据缺失场景**：

|缺失证据|影响|
|---|---|
|毛利数据缺失|审核结论可信度降为 missing\_data，只能 preview|
|销量数据缺失|审核结论可信度降为 missing\_data，只能 preview|
|客群基线未完成|客群匹配维度无证据，相关判断只能 preview|
|关键判断依据无证据引用（如"客群不匹配"判断无证据）|该条结论只能 preview|
|规则引用为空（rule\_refs 为空）|该条建议可信度最高为 low，报告标记 preview|

**作战卡不能进入分配或关闭的证据缺失场景**：

|场景|限制|
|---|---|
|作战卡分配前|关联审核结论必须为 formal（非 preview/blocked），否则禁止分配|
|作战卡关闭前|必须检查：执行回传已提交、执行证据已上传、执行后指标或缺数据说明已填写，缺一不可关闭|

**Combo Card 审核报告导出前检查项**：

- 毛利数据是否齐全

- 销量数据是否齐全

- 客群基线是否完成

- 规则引用（rule\_refs）是否齐全

- 关键判断依据证据是否齐全

> 任一缺失 → 报告只能导出 `preview_report`，不得导出 `formal_report`，并显示 blocked 原因清单。

**作战卡关闭前检查项**：

- 执行回传是否已提交

- 执行证据是否已上传

- 执行后指标是否填写，或是否提供缺数据说明

> 任一缺失 → 作战卡不可关闭，系统显示 blocked 原因。

**blocked 原因展示**：证据缺口未补齐时，系统必须在审核结论页、作战卡详情页、报告导出页明确显示 blocked 原因清单（缺口项 \+ 缺什么 \+ 补齐路径）。

## 外部集成

### LLM 判断边界（总则）

LLM 的输出不能直接等同于事实或专家规则。所有调用 LLM 的维度（客群匹配、场景匹配、出品落地、Combo Card 审核等）必须遵守以下边界：

- **结构化输出**：LLM 只能生成结构化候选判断，最终结果必须经过 schema 校验，校验失败不得直接采用。

- **必含字段**：LLM 输出必须包含 `evidence_ids`（证据引用）、`rule_refs`（规则引用）、`confidence`（可信度）、`missing_data`（缺失数据）。

- **可信度约束**：没有证据或规则引用时，只能输出"待确认"，不得给 high confidence（最高为 low）。

- **优先级**：规则卡和红线卡优先级高于 LLM 文案判断；LLM 不得推翻已命中红线卡的禁止动作。

- **失败回退**：LLM 失败、超时、超预算时，系统必须回退到规则兜底或 dry-run，不得直接报错中断主流程。

- **调用留痕**：所有 LLM 调用需记录输入摘要、输出、schema 校验结果、费用和错误原因。

### DeepSeek适配器

**封装结构**：

```Python

    def __init__(self, config: LLMConfig):
        self.api_url = config.api_url      # OpenAI-compatible endpoint
        self.api_key = config.api_key
        self.model = config.model_name
        self.dry_run = config.dry_run      # True时返回mock结果

    async def structured_call(
        self,
        prompt: str,
        output_schema: dict,    # JSON Schema，必须包含 evidence_ids/rule_refs/confidence/missing_data
        temperature: float = 0.3
    ) -> dict:
        """
        调用LLM并校验输出格式。
        - dry_run=True时返回schema的mock数据
        - 响应不符合schema时重试1次
```

### 表关系

```Plaintext
store ──1:1── store_baseline
store ──1:N── store_redline
store ──1:N── dish
store ──1:N── combo
store ──1:N── diagnosis_result
store ──1:N── menu_health
store ──1:N── combo_card_audit

dish ──1:N── dish_sales
dish ──1:N── combo_card_audit (level=dish_item)
dish ──1:N── battle_card

combo ──N:M── dish (via combo_dish)
combo ──1:N── combo_card_audit (level=combo_overall / dish_item，主对象)

combo_card_audit ──1:N── battle_card

battle_card ──1:N── execution_report (含版本号，多次提交保留版本)
battle_card ──1:1── review_result
battle_card ──1:N── evidence (via ref_type/ref_id)
battle_card ──1:N── h5_access_log

evidence ──ref── diagnosis_result / combo_card_audit / battle_card / execution_report
feedback ──ref── combo_card_audit / execution_report
```

## 交付里程碑

|**阶段**|**交付内容**|**闭环能力**|
|---|---|---|
|M1 基础数据|F01 \+ F02 \+ 数据库 \+ Web 后台骨架|数据录入和门店基线可运行|
|M2 诊断引擎|F03 \+ F04 \+ LLM 适配器 \+ FastGPT 适配器|菜品诊断和健康度可输出|
|M3 审核与作战|F05 \+ F06 \+ 证据体系（E01\-E04）|审核结论和作战卡可生成|
|M4 执行闭环|F07 \+ F08 \+ H5 门店端 \+ 企微 Bot|执行回传和复核可完成|
|M5 反馈回写|F09 \+ F10 \+ 报告导出|全链路闭环可跑通|
|M6 集成与稳定|OCR 适配器 \+ POS 适配器 \+ Bug 修复 \+ 性能优化|生产可用|

每个里程碑交付时，需跑通对应阶段的闭环验收场景。
