# 产品需求方案 V2\.0 版本（开发前最终版）

> 版本变更说明
>
> V2.0 相对 V1.1 为产品形态级重构，是进入开发前的最终需求文档。核心变化：
>
> 1. **主数据全部从 SaaS 只读获取，不持久化**：菜品/成本/销量/套餐构成/门店基础信息均从 SaaS（五乡 cysms 系统）只读获取，本系统不回写、不落库。SaaS 接口包装为智能体工具（skill），对话时按需调用，聚合结果走 Redis 短期缓存。移除 CSV 导入、OCR 解析、手动录入、POS 适配器、本地主数据表与全量/增量同步任务，门店的客群基线，品牌红线，套餐审核配置是不是还需要配置？
> 2. **诊断 / 健康度 / 套餐审核全部融入智能体对话**：不再设独立诊断中心页面。运营与智能体对话，智能体在对话中完成菜品多维诊断、菜单健康度、Combo 套餐级审核，并直接产出优化建议。
> 3. **作战卡 = 对话生成的优化建议**：智能体在对话中产出优化建议（即作战卡），运营从优化建议生成可执行**任务**，任务走分配 → 执行回传 → 复核闭环。
> 4. **证据中心取消，改为智能体溯源**：证据不再是独立管理的实体，而是智能体回复过程中的溯源信息（知识库引用 / SaaS 接口调用 / 网络查询）。溯源信息实时展示在对话中，并持久化存储供后续追溯。
> 5. **保留 V1.1 已确认的工程边界**：规则卡 / 案例卡 / 红线卡结构化字段、LLM 判断边界收紧、门店端授权安全规则（iLink 身份绑定）、任务关闭 Gate、溯源完整性 Gate。
>
> 6. **门店端为微信小程序**（V2.0 修正）：店长通过微信登录小程序查看与执行任务，取消原 H5 token 临时链接机制，改为小程序登录态 \+ 任务归属授权。
>
> 7. **去掉企业微信，改用微信 iLink Bot 接入**（V2.0 修正）：企微安装运营成本高，改为通过腾讯官方微信 iLink Bot API（ClawBot/OpenClaw）把自研智能体接入微信个人号。老板在微信直接和 AI 助手对话拍板，实现「微信即智能体」多端统一对话。详见《微信 iLink Bot 接入与多端对话详细设计》。
>
> 8. **这一期不做小程序，店长也走 iLink 微信**（V2.0 修正）：门店端小程序推后。店长在微信加 AI 助手号，接收任务通知并以对话方式回传执行情况（发图片 \+ 文字，如"已执行，这周 31 份毛利 55%"），智能体解析为结构化执行回传。老板 / 店长都走微信，运营走 Web 后台，架构统一、零安装。
>
> 关键假设（开发前需确认，文档内以 `【待确认】` 标注）：
>
> - SaaS 除流水接口外是否提供菜品主数据字典、门店字典独立接口；当前以 `getserialdata` 流水接口为主，由智能体工具实时聚合 \+ Redis 缓存，预留字典接口扩展位。
> - 套餐审核所需的业务标注字段（目标客群 / 老板目标 / 平台团购标记 / 午市晚市）SaaS 不提供，由本系统作为"套餐审核配置"维护，不回写 SaaS。
> - 门店定位标签 / 主力时段 / 客单价区间 / 客群基线 / 品牌红线，SaaS 不提供，由本系统维护。

# 产品定位

1. 为精品正餐品牌提供菜单决策闭环系统。以智能体对话为核心交互，从 SaaS 数据只读接入到执行回写，让每一次换菜、下架、改规格都有依据、有溯源、有跟踪、有反馈。

2. 核心价值：把凭经验调菜单变成有数据、有规则、有溯源、有闭环的菜单决策。智能体负责诊断与建议，人负责拍板与执行。

# 闭环主链路

```Plaintext
SaaS 数据只读拉取 → 门店配置（客群基线 / 品牌红线 / 套餐审核配置）
→ 智能体对话（菜品诊断 + 菜单健康度 + Combo 套餐级审核 + 优化建议，带溯源）
→ 优化建议（作战卡）→ 生成任务 → 任务分配（iLink Bot 推送至店长微信）
→ 门店执行回传（店长微信对话回传，含图片 + 文字，智能体解析为结构化回传）
→ 执行复核 → 老板反馈 → 规则 / 案例 / 红线回写
```

# 功能详设

## SaaS 数据只读接入

**说明**：所有主数据从 SaaS（五乡 cysms 系统）只读获取，本系统不回写 SaaS。**不持久化主数据，不做全量/增量同步任务**。SaaS 接口包装为智能体工具（skill），对话时按需调用，聚合结果走 Redis 短期缓存。详见《SaaS 只读接入详细设计》。

**数据来源**：SaaS 流水接口 `POST https://cysms.wuuxiang.com/api/datatransfer/getserialdata`

**鉴权**：Header `access_token` / `accessid` / `granttype`

**核心请求参数**：

|参数|必填|说明|
|---|---|---|
|centerId|是|集团/品牌 ID|
|shopId|是|门店 ID|
|settleDate / beginDate / endDate|是|结算日期或日期区间，`yyyy-MM-dd`|
|pageNo / pageSize|是|分页，pageSize 上限 500|
|dateType|否|日期类型（1 营业日 / 2 自然日等）|
|needPkgDetail|否|1 返回套餐明细（拉套餐必传）|

**接口返回**：`billList[]`，每条账单含：

- 账单级：`bs_id` / `bs_code` / `shop_id` / `shop_code` / `shop_name` / `brand_name` / `center_name` / `open_time` / `settle_time` / `people_qty` / `order_type` / 会员信息
- `item[]` 消费明细：`item_id` / `item_name` / `item_code` / `cost_price`(成本价) / `last_price`(售价) / `orig_price` / `last_qty`(数量) / `last_total` / `small_class_name` / `big_class_name`(分类) / `unit_name`(单位) / `pkg_flg`(套餐标记 0/1/2) / `pkg_sc_id`(套餐主菜品 id) / `sales_revenue_type_id`
- `settleDetail[]`(支付) / `discountDetail[]`(折扣) / `promoteDetail[]`(促销) / `fulloffDetail[]`(满减)

### 智能体工具（SaaS Skills）

SaaS 接口包装为智能体可调用工具，对话时按需调用，实时从流水聚合（不落库），结果走 Redis 缓存：

|工具|出参|缓存 TTL|
|---|---|---|
|`get_dishes`|菜品视图（distinct item \+ 分类 \+ 售价众数 \+ 成本）|24h|
|`get_sales`|销量序列（sum last\_qty by 周期，套餐明细不重复计）|4h|
|`get_combos`|套餐构成 \+ 价格（pkg\_flg/pkg\_sc\_id 识别）|24h|

**缓存策略**：Redis 按 `saas:{type}:{store_id}:{params_hash}` 缓存；运营可手动刷新（清缓存）；单门店拉取加锁防击穿。

**降级**：SaaS 不可用时缓存命中仍可服务；缓存也 miss 则该维度标 `missing_data`，结论降级 preview（溯源完整性 Gate 兜底）。

**只读视图页面**：门店详情内"菜品数据（SaaS 只读）"标签页实时调工具（带缓存）展示菜品/成本/销量/套餐，不可编辑。

**API**：

```Plaintext
GET  /api/v1/stores/{id}/saas/dishes    # 菜品只读视图（实时，走工具+缓存）
GET  /api/v1/stores/{id}/saas/combos    # 套餐只读视图（实时）
POST /api/v1/stores/{id}/saas/refresh   # 手动刷新缓存（运营）
GET  /api/v1/stores/{id}/saas/health    # SaaS 可达性与缓存状态
```

> 【待确认】SaaS 是否提供独立的菜品主数据字典 / 门店字典接口。若有，工具优先调字典；若无，继续用流水聚合。

## 智能体对话

**页面**：智能体对话 → 对话工作台

**核心交互**：运营与智能体多轮对话，智能体在对话中完成菜品多维诊断、菜单健康度、Combo 套餐级审核，并产出优化建议（作战卡）。所有结论附带溯源信息。

### 对话上下文

- 运营选择门店作为对话上下文（一个门店可有多个对话会话）
- 可选择分析范围：全店 / 指定套餐 / 指定菜品
- 对话带历史，支持多轮追问
- 智能体调用 SaaS 数据（只读聚合视图）\+ 客群基线 \+ 品牌红线 \+ 知识库（规则卡/案例卡/红线卡）\+ 网络查询

### 交互流程

1. 运营选择门店，新建或打开对话

2. 运营输入需求，如"帮我分析这个门店的菜单健康度"、"审核一下双人商务套餐"

3. 智能体调用 SaaS 数据与知识库，在对话中输出：
   - 诊断结论（菜品多维诊断 / 菜单健康度 / 套餐级审核）
   - 优化建议卡片（作战卡）
   - 溯源信息（引用了哪些知识库规则 / SaaS 数据 / 网络来源）

4. 运营可追问、要求调整建议、要求对比方案

5. 运营确认优化建议 → 一键生成任务

### 智能体对话中的诊断能力

以下能力均为智能体对话内部能力，不在前端独立设页，结论以对话消息形式输出。

#### 菜品多维诊断（7 维）

智能体基于 SaaS 聚合数据计算，规则同 V1.1：

- **毛利维度**：单品毛利率 = (售价 \- 成本) / 售价；标签 高/中/低毛利 / 低于红线 / 数据缺失
- **销量/偏好维度**：同分类内销量排名 前20%/中60%/后20%；趋势 上升/平稳/下降/待定
- **品牌角色维度**：优先人工标注，未标注按销量×毛利自动推断（招牌/流量/利润/陪衬/常规）
- **客群匹配维度**：调用 LLM，输入菜品 \+ 客群基线 \+ 品牌定位；受 LLM 判断边界约束
- **场景匹配维度**：调用 LLM，输入菜品 \+ 主力消费场景 \+ 时段；受 LLM 判断边界约束
- **出品落地维度**：优先人工标注，未标注调用 LLM 推断 易/中/难
- **组合贡献维度**：核心搭配 / 可替换 / 无搭配（基于套餐构成）

> 所有 LLM 维度输出必须含 `evidence_ids` / `rule_refs` / `confidence` / `missing_data`，无证据/规则引用不得给 high confidence（详见"LLM 判断边界"）。

#### 菜单健康度

智能体计算总分与各维度得分，输出风险列表与修复建议。

```Plaintext
总分 = 结构得分 × 0.25 + 价格带得分 × 0.20 + 毛利得分 × 0.30 + 招牌保护得分 × 0.25
```

各维度计算规则同 V1.1（结构 / 价格带 / 毛利 / 招牌保护，满分 100）。风险规则（毛利风险 / 结构风险 / 品牌风险 / 价格带风险）同 V1.1。

#### Combo 套餐级审核

审核主对象为 `combo_id`（套餐）。智能体在对话中输出套餐整体结论 \+ 单菜调整建议。

**套餐整体字段**：目标套餐价、套餐成本、套餐毛利率、平台团购标记、午市/晚市、目标客群、老板目标、是否包含招牌菜（构成来自 SaaS 只读，业务标注来自套餐审核配置）。

**套餐级风险**：毛利风险、品牌/客群风险、出品落地风险、组合价值风险。

**单菜明细风险**：建议替换 / 建议改规格 / 必须老板拍板。

**输出结构**（对话消息中结构化呈现，同时保留套餐整体结论与单菜调整建议）：

**可信度评级规则**：

|**可信度**|**条件**|
|---|---|

|low|\<= 4 维数据完整，或 LLM 判断无证据/规则引用|
|missing\_data|关键维度（毛利或销量）数据缺失|

**老板拍板项标记规则**：

- 建议为"下架招牌菜" → 必须标记拍板
- 建议依据中可信度为 low → 标记拍板
- 多条建议互斥 → 标记拍板
- 证据缺口数量 \> 维度数量的 50% → 标记拍板
- 套餐整体结论与老板目标冲突 → 标记拍板

**交互流程**：

1. 运营在对话中确认某条优化建议（作战卡）

**溯源完整性 Gate（分配前）**：作战卡关联的溯源必须完整（数据完整 \+ 规则引用齐全 \+ 关键判断依据有溯源），否则禁止分配，系统显示 blocked 原因。

## 任务分配与门店执行回传

**说明**：V2.0 这一期不做小程序。店长通过微信 iLink Bot 接收任务通知，并以对话方式回传执行情况（发图片 \+ 文字），智能体负责将自然语言 \+ 图片解析为结构化执行回传。

**触达方式**：

- 运营在 Web 后台分配任务后，系统通过 iLink Bot 主动推送任务通知到店长微信
- 推送消息包含：任务动作、目标菜品/套餐、预期效果、证据需求、观察周期
- 店长直接在微信对话中回复文字 \+ 图片完成执行回传
- 无需安装任何 App 或小程序，零安装成本

**身份与授权**：

- 店长首次加 AI 助手号为微信好友后，回复手机号完成身份绑定（iLink userId ↔ store\_staff 账号）
- 系统按 iLink userId 识别店长身份，关联到所属门店和已分配任务
- 店长只能查看/操作分配给自己的任务，越权查询返回"未找到相关任务"
- 运营可在后台停用店长账号（status=disabled），停用后该店长消息不再被识别为有效回传
- 任务分配时绑定店长账号（store\_staff\_id），未绑定店长的任务不推送

**交互流程（对话式）**：

1. 运营在 Web 后台分配任务 → iLink Bot 推送到店长微信
2. 推送示例："【新任务】门店：XX 店，动作：替换 麻婆豆腐 → 蟹粉豆腐，预期效果：提升毛利率 28%→55%，观察周期：14 天。请执行后回复。"
3. 店长执行完毕，在微信回复：
   - 文字：如"已执行，撤掉了麻婆豆腐换蟹粉豆腐，这周 31 份，毛利 55%，比之前好不少"
   - 图片：上传菜单照片、后厨出品照片等（直接通过微信发图片）
4. 智能体解析店长自然语言回复为结构化执行回传：
   - 提取执行状态（已执行 / 未执行 / 无法执行）
   - 提取执行后指标（销量、毛利率等数值）
   - 提取未执行原因（如适用）
5. 系统记录结构化回传（保留版本快照）
6. 智能体回复店长"已收到回传，已记录"——确认闭环

**智能体解析规则**：

- 执行状态识别：关键词"已执行/执行了/换了/改了"→ executed；"没执行/做不了/无法"→ not\_executed/cannot\_execute
- 数值提取：正则匹配"XX 份""毛利 XX%""毛利率 XX%""XX 元"等模式
- 图片关联：微信图片自动下载存储，关联到 task\_execution 记录
- 解析失败兜底：若智能体无法解析，标记"待人工复核"，推送运营处理

**iLink 对话回传数据映射**（智能体解析输出 → task\_execution 表）：

|**智能体解析字段**|**对应 DB 字段**|**必填**|**说明**|
|---|---|---|---|
|执行状态|execution\_status|是|executed/not\_executed/cannot\_execute|
|执行人|executor\_name|是|默认取店长姓名（从绑定账号取）|
|执行日期|execution\_date|是|取消息发送日期|
|实际动作|actual\_action|已执行时|从文字提取具体操作描述|
|未执行原因|not\_executed\_reason|未执行时|从文字提取原因|
|图片列表|execution\_evidence|已执行时|从微信消息图片下载，校验类型/大小|
|执行后指标\-观察周期|observation\_period|否|从文字提取周期数值|
|执行后指标\-结果值|result\_value|否|从文字提取数值|

**图片安全规则**：

- 微信消息中的图片通过 iLink 接口下载，校验文件类型（仅允许 jpg/png，通过 magic bytes）
- 限制单张大小 \< 5MB，单次回传最多 5 张
- 图片存储到本地文件系统（或对象存储），数据库只记录映射后的访问 URL
- 文件路径不对外暴露

**业务规则**：

- 提交人和提交时间由系统自动记录（取 iLink 绑定身份）
- 同一任务可多次提交回传（补充证据），每次提交保留版本快照，不覆盖；执行状态以最新一次为准，历史版本可追溯
- 智能体解析失败时标记"待人工复核"，运营在复核页面手动补录

**访问日志**：每次推送、回传、图片上传均记录日志（weixin\_user\_id / action / ip / success / error\_reason）。

**API**：


**说明**：取代原"证据中心"。溯源是智能体回复过程中的来源信息，不再是独立管理的实体。溯源信息实时展示在对话消息中，并持久化存储供后续追溯。

- 每条智能体回复消息附带 `trace_ids` 列表
- 溯源信息随消息持久化存储，后续可回溯每条建议引用了哪些知识库/接口/网络来源
- 对话界面中每条结论可展开查看溯源详情

### 溯源完整性 Gate

溯源完整性作为正式交付的硬性 Gate，缺口未补齐时系统必须明确显示 blocked 原因。

**只能生成 preview、不能正式交付的溯源缺失场景**：

|缺失溯源|影响|
|---|---|
|毛利数据缺失（SaaS 未同步）|结论可信度降为 missing\_data，只能 preview|
|销量数据缺失|结论可信度降为 missing\_data，只能 preview|
|客群基线未完成|客群匹配无溯源，相关判断只能 preview|
|关键判断无溯源引用|该条结论只能 preview|
|规则引用为空（rule\_refs 为空）|该条建议可信度最高为 low，标记 preview|

**任务分配前检查**：作战卡关联溯源必须完整，否则禁止分配。

**任务关闭前检查**：执行回传已提交、图片已上传、执行后指标或缺数据说明已填写，缺一不可关闭。

**报告导出前检查**：毛利数据、销量数据、客群基线、规则引用、关键判断依据溯源齐全，任一缺失只能导出 preview\_report。

**blocked 原因展示**：溯源缺口未补齐时，系统在对话消息、任务详情、报告导出页明确显示 blocked 原因清单。

# 外部集成

## LLM 判断边界（总则）

LLM 的输出不能直接等同于事实或专家规则。所有调用 LLM 的维度（客群匹配、场景匹配、出品落地、套餐审核等）必须遵守：

- **结构化输出**：LLM 只能生成结构化候选判断，最终结果必须经过 schema 校验，校验失败不得直接采用
- **必含字段**：LLM 输出必须包含 `evidence_ids`、`rule_refs`、`confidence`、`missing_data`
- **可信度约束**：没有证据或规则引用时，只能输出"待确认"，不得给 high confidence（最高为 low）
- **优先级**：规则卡和红线卡优先级高于 LLM 文案判断；LLM 不得推翻已命中红线卡的禁止动作
- **失败回退**：LLM 失败、超时、超预算时，系统必须回退到规则兜底或 dry-run，不得直接报错中断主流程
- **调用留痕**：所有 LLM 调用需记录输入摘要、输出、schema 校验结果、费用和错误原因

## 微信 iLink Bot 接入

通过腾讯官方微信 iLink Bot API（ClawBot/OpenClaw）把自研智能体接入微信个人号，老板/店长在微信直接和 AI 助手对话。零客户安装成本，取代企业微信 Bot。详细设计见《微信 iLink Bot 接入与多端对话详细设计》。

**多端统一对话架构**：一个智能体后端，三端触达。

|端|通道|体验|
|---|---|---|
|运营|Web 对话工作台|诊断/审核/生成任务/复核|
|店长|微信对话（iLink Bot）|接收任务推送、对话式执行回传（发图片+文字）|
|老板|微信对话（iLink Bot）|拍板、问情况、回复反馈|

**主动推送场景**（iLink `sendmessage`）：

|场景|推送对象|内容|
|---|---|---|
|优化建议通知|老板/运营|门店 \+ 风险数 \+ 建议摘要|
|任务分配通知|店长|动作 \+ 目标菜品 \+ 观察周期 \+ 证据需求 \+ 回复引导|
|执行回传提醒|店长|任务 X 距截止 N 天，请回传（\"回复图片+文字即可\"）|
|老板拍板请求|老板|建议内容 \+ 回复引导（\"回复同意/不同意/修改意见\"）|

**老板拍板交互**（对话式）：

1. 系统通过 iLink 主动推送拍板请求到老板微信
2. 老板在微信直接回复自然语言（"同意"/"不同意"/"换小份"）
3. 智能体解析老板回复为决策（agree/disagree/revision）
4. 系统记录老板决策，更新作战卡/任务状态，回复老板"已收到"

**身份绑定**：老板/店长首次在微信加 AI 助手号发消息 → 回复手机号 → 绑定 userId ↔ 本系统账号。

```Plaintext
store ──1:1── store_baseline

store ──1:N── store_staff

store ──1:N── case_card (via store_id)
store ──1:N── brand_knowledge (via store_id, NULL=品牌级)

SaaS 主数据（菜品/销量/套餐）不持久化，由智能体工具按需调用 SaaS \+ Redis 缓存
combo_config.saas_combo_key ──关联── SaaS 工具聚合的套餐结果

conversation ──1:N── message
message ──1:N── message_trace
conversation ──1:N── battle_card

battle_card ──1:N── task
store_staff ──1:N── task (via assignee_staff_id)

boss ──1:1── ilink_binding (via ref_id, role=boss)

task_execution ──1:N── execution_evidence
task ──1:1── review_result
task ──1:N── ilink_access_log

feedback ──ref── battle_card / task_execution
```

# 交付里程碑

|**阶段**|**交付内容**|**闭环能力**|
|---|---|---|
|M1 SaaS接入与门店配置|SaaS 只读适配器 \+ 智能体工具(skill)\+ Redis 缓存 \+ 门店配置（客群基线/品牌红线/套餐审核配置）\+ 数据库 \+ Web 后台骨架|SaaS 数据可只读拉取（工具+缓存），门店配置可运行|
|M2 智能体对话|对话工作台 \+ LLM 适配器(判断边界) \+ FastGPT 适配器 \+ 溯源体系|对话中可输出诊断/健康度/套餐审核结论与优化建议，带溯源|
|M3 优化建议与任务|作战卡(优化建议) \+ 任务生成/分配 \+ 溯源完整性 Gate|优化建议可转为任务并分配|
|M4 执行闭环|微信 iLink Bot 接入(含身份绑定/主动推送/Webhook) \+ 店长账号体系 \+ 智能体对话式回传解析(图片/文字→结构化) \+ 执行回传(图片/版本/授权安全) \+ 复核|店长微信可接收任务并对话回传，老板微信可拍板，执行复核可完成|
|M5 反馈回写|老板反馈 \+ 规则/案例/红线回写 \+ 报告导出|全链路闭环可跑通|
|M6 稳定与优化|Bug 修复 \+ 性能优化 \+ SaaS 字典接口接入(若确认)|生产可用|

每个里程碑交付时，需跑通对应阶段的闭环验收场景。

> 说明：OCR / CSV / 手动录入 / POS 适配器已在 V2.0 移除，主数据统一由 SaaS 只读接入。
