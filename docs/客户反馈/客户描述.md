# OmniSight｜开发团队对齐说明 v1.2（最终版）

## 0. 一句话定义
OmniSight 是一个以“菜单”为入口的餐厅经营决策系统，通过门店真实数据 + 商圈上下文 + 人工校准 + AI分析，生成7天作战卡，并通过执行证据、ROI账本和知识库回写形成闭环。

## 1. 我们要做什么
不是SaaS / BI / 报表 / 咨询系统，而是：
把餐厅经营事实 → 转成可执行动作 → 持续优化经营决策

核心问题：
- 数据多但无法决策
- 不知道原因
- 调整依赖经验
- 咨询不持续
- BI无动作闭环

## 2. 系统闭环
菜单 → 诊断 → 机会点 → 作战卡 → 执行 → 证据 → ROI → 知识库

## 3. 核心结构

### 品牌知识库
菜单/菜品/规则/老板偏好/历史卡/ROI

### 门店上下文画像（关键）
门店不是数据点，而是：
- 商圈
- 客群
- 竞品
- 执行能力
- 菜单约束
- 地理位置（300m/800m/3km）

### 数据层（3类）
内部数据 + 外部数据 + 人工校准

### 菜单诊断Agent
四象限 + 三维评分 + 机会点 + 作战卡候选

### 作战卡系统
7天实验单元：
问题 → 动作 → 负责人 → 指标 → 执行证据 → ROI → 复盘

### 状态机
待确认→执行→证据→复核→ROI→回写

### 证据系统
照片/截图/推荐记录/POS

### ROI系统
执行分/效果分/复用分（0-3）

### 知识库回写
形成长期经营记忆

## 4. 微信系统
微信只是入口：
老板 / 店长 / 区域 / 内部

事件驱动：
card创建→确认→执行→上传→复核→ROI→回写

## 5. P0开发范围
6个页面：
1作战卡列表
2老板确认
3门店执行
4证据上传
5ROI账本
6知识库

## 6. 测试店节奏
1-5家：
第0周建画像
第1周3张卡
第2周ROI
第3周跨店
第4周复盘

## 7. 原则
不做BI / SaaS / 咨询 / 聊天系统

## 8. 定义
OmniSight = 菜单 + 上下文 + AI + 作战卡 + 证据 + ROI + 知识库闭环

## 9. 一句话
我们不是工具，是经营动作系统
