# 技术需求 V1.0 反馈：V1.1 需补充项

> 说明：以下反馈仅针对当前 V1.0 需求方案需要补充的产品与工程边界。OCR / POS 接入里程碑暂不要求调整，按现有接口文档和接入条件继续推进。

## 1. Combo Card 审核需要升级为“套餐级审核”

当前方案里的 Combo Card 审核主要按单个菜品触发和输出，但我们的核心场景是套餐 / Combo Card 审核，不能只停留在单菜维度。

请在 V1.1 中补充：

- `combo_id` 或套餐对象作为审核主对象。
- 套餐整体字段：目标套餐价、套餐成本、套餐毛利率、平台团购标记、午市 / 晚市、目标客群、老板目标、是否包含招牌菜。
- 套餐级风险：毛利风险、品牌 / 客群风险、出品落地风险、组合价值风险。
- 单菜明细风险：哪些菜建议替换、哪些菜建议改规格、哪些菜必须老板拍板。
- 输出结构中同时保留“套餐整体结论”和“单菜调整建议”。
- 数据模型中不要只用 `dish_id` 表达审核对象，应能关联 `combo_id` 和多个菜品。

## 2. 规则卡 / 案例卡 / 红线卡字段需要补全

当前规则、案例、红线字段偏薄，容易变成普通文本知识库，不足以支撑 ExpertOS 的可调用判断资产。

请在 V1.1 中补充：

- 规则卡必须包含：触发条件、默认判断、例外条件、风险类型、所需数据、迁移边界、对应输出字段、证据引用、可信度、版本号。
- 案例卡必须包含：适用场景、关联规则、执行动作、执行前数据、执行后数据、结果判断、可复用结论、不可迁移边界。
- 红线卡必须包含：触发条件、禁止动作、必须动作、老板拍板条件、适用范围、例外情况、证据要求。
- FastGPT 检索结果只能引用这些结构化资产，不能替代规则卡本身。

## 3. 证据链需要增加“正式交付 Gate”

当前方案已有 `preview / formal / blocked`，但还需要明确什么情况下不能正式交付。

请在 V1.1 中补充：

- 哪些证据缺失时只能生成 `preview`，不能生成正式报告。
- 哪些证据缺失时作战卡不能进入分配或关闭。
- Combo Card 审核报告导出前，需要检查：毛利数据、销量数据、客群基线、规则引用、关键判断依据证据。
- 作战卡关闭前，需要检查：执行回传、执行证据、执行后指标或缺数据说明。
- 报告导出接口需要区分 `preview_report` 和 `formal_report`。
- 如果证据缺口未补齐，系统必须明确显示 blocked 原因。

## 4. H5 执行回传链接和证据上传需要补安全规则

当前 H5 临时链接和图片上传流程可以作为 P0，但需要补最小安全边界，避免后续真实门店使用时出问题。

请在 V1.1 中补充：

- token 只能绑定单张作战卡，不能访问其他门店或其他作战卡。
- token 必须支持过期、撤销和重新生成。
- 过期 token 只能查看提示，不能继续提交。
- 每次打开、提交、上传证据都要记录访问日志。
- 同一作战卡多次提交时，需要保留提交版本，不要只覆盖最后一次。
- 上传文件需要限制类型、大小、数量，并校验真实文件类型。
- 上传文件路径不能直接暴露本地绝对路径。

## 5. LLM 判断边界需要收紧

当前客群匹配、场景匹配、出品落地等维度会调用 LLM，但 LLM 的输出不能直接等同于事实或专家规则。

请在 V1.1 中补充：

- LLM 输出必须包含：`evidence_ids`、`rule_refs`、`confidence`、`missing_data`。
- 没有证据或规则引用时，只能输出“待确认”，不能给 high confidence。
- 规则卡和红线卡优先级高于 LLM 文案判断。
- LLM 只能生成结构化候选判断，最终结果必须经过 schema 校验。
- LLM 失败、超时、超预算时，系统必须回退到规则兜底或 dry-run。
- 所有 LLM 调用需要记录输入摘要、输出、schema 校验结果、费用和错误原因。
