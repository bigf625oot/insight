# 优化建议转任务与执行闭环详细设计 V1\.0

> 本文档是《产品需求方案 V2\.0》中"优化建议转任务 / 任务执行回传 / 复核"模块的开发前详细设计。定义作战卡(优化建议)生命周期、任务生成与状态机、溯源完整性 Gate、关闭 Gate、店长 iLink 身份绑定与对话式回传、回传版本管理、图片上传安全、智能体自然语言解析、复核与老板拍板联动、异常边界与接口契约。

# 1\. 设计目标

1. **建议可执行**：智能体对话产出的优化建议（作战卡）能一键转化为可分配、可执行、可回传、可复核的任务。
2. **Gate 可控**：溯源不完整的建议不可分配；证据不齐的任务不可关闭，全程 blocked 原因可见。
3. **安全可控**：店长通过 iLink 微信 Bot 身份绑定接入、任务归属校验、访问留痕；图片上传校验真实类型、路径不暴露。
4. **版本可溯**：执行回传多次提交保留版本快照，历史可追溯。
5. **闭环可回写**：复核结果与老板拍板联动规则/案例/红线回写。

# 2\. 作战卡（优化建议）生命周期

## 2\.1 状态流转

```Plaintext
智能体生成 → pending(待确认)
   │
   ├─(运营确认)──→ confirmed(已确认) ──→ 可生成任务
   ├─(运营驳回)──→ rejected(已驳回)
   ├─(运营要求调整)──→ 新版本生成，旧版 superseded
   └─(需老板拍板)──→ needs_boss_decision(待拍板)
                          │
                    (老板拍板后)──→ confirmed / rejected
```

## 2\.2 operator\_status 枚举

|状态|说明|可执行动作|
|---|---|---|
|`pending`|智能体生成，待运营确认|确认/驳回/调整/发老板拍板|
|`confirmed`|运营或老板确认|生成任务|
|`rejected`|运营或老板驳回|归档|
|`needs_boss_decision`|需老板拍板|iLink 推送拍板请求到老板微信|
|`superseded`|被新版本替代|仅可查看|

## 2\.3 版本管理

- "要求调整"生成新 `battle_card` 记录，旧记录 `superseded_by` 指向新记录 ID。
- 同一对象（combo\_view\_id / saas\_item\_id）的最新非 superseded 版本为有效建议。
- 右栏建议清单只展示有效版本，历史版本可折叠查看。
- 已 confirmed 的建议被调整：原 confirmed 保留，新版本重新走确认流程（不自动继承确认）。

## 2\.4 字段映射（作战卡 → 任务）

生成任务时，以下字段自动从 battle\_card 映射到 task：

|task 字段|来源|
|---|---|
|`battle_card_id`|battle\_card.id|
|`store_id`|battle\_card.store\_id|
|`action_type`|battle\_card.suggestion\_type|
|`action_desc`|battle\_card.suggestion\_desc|
|`expected_effect`|battle\_card.expected\_effect|
|`rule_sources`|battle\_card.rule\_refs \+ case\_refs|
|`evidence_requirements`|按 action\_type 预置默认证据需求（见 2\.5）|

运营可在生成后编辑：`assignee` / `observation_period_days` / `evidence_requirements`。

## 2\.5 默认证据需求（按动作类型）

|action\_type|默认证据需求|
|---|---|
|换菜|新菜单照片；替换后 1 周新菜销量；替换后毛利率|
|下架|下架后菜单照片；原位置销量变化|
|改规格|新规格照片；改规格后 2 周销量；成本变化|
|新增|新菜品照片；上市 2 周销量|
|调价|新价签/菜单照片；调价后 2 周销量；毛利率变化|

# 3\. 任务状态机

## 3\.1 状态流转图

```Plaintext
                ┌─────────────┐
                │ pending_assign│ ← 生成任务
                └──────┬──────┘
                       │ (运营分配门店+执行人)
                       ↓
                ┌─────────────┐
        ┌──────│   pending   │ ← 待执行
        │       └──────┬──────┘
        │              │ (店长首次打开)
        │              ↓
        │       ┌─────────────┐
        │       │ in_progress │ ← 执行中
        │       └──────┬──────┘
        │              │
        │       ┌──────┴──────┐
        │       ↓             ↓
        │ ┌──────────┐ ┌──────────────────┐
        │ │ completed│ │ cannot_execute   │
        │ └──────────┘ └──────────────────┘
        │  (店长回传)    (店长回传+原因)
        │
        └─ (运营撤销分配) → 回到 pending_assign
```

## 3\.2 流转条件与 Gate

|流转|触发|前置 Gate|失败处理|
|---|---|---|---|
|pending\_assign → pending|运营分配店长|**溯源完整性 Gate**（见 4）|Gate 未过禁止分配，返回 GATE\_BLOCKED|
|pending → in\_progress|店长微信查看任务（iLink 推送后首次对话交互）|iLink 身份绑定有效 \+ 任务归属校验|未绑定/越权返回拒绝|
|in\_progress → completed|店长微信回传（智能体解析自然语言为结构化回传）|无（回传即流转，关闭 Gate 在"关闭"环节）|\-|
|in\_progress → cannot\_execute|店长回传"无法执行"+原因（智能体解析）|无|\-|
|completed/cannot\_execute → 关闭|运营复核完成|**关闭 Gate**（见 5）|Gate 未过不可关闭|

## 3\.3 特殊流转

- **pending\_assign → pending\_assign（撤销分配）**：运营可撤销已分配任务，回到待分配（解除店长绑定）。
- **cannot\_execute → pending\_assign（重新分配）**：无法执行的任务，运营调整后可重新分配（重新绑定店长）。
- **completed/cannot\_execute 不可回退到 in\_progress**：终态。需重做则新建任务。

## 3\.4 任务关闭定义

"关闭"指任务进入终态后运营完成复核、写入 review\_result，任务彻底结束。关闭 Gate 在此环节校验。

# 4\. 溯源完整性 Gate（分配前）

## 4\.1 检查项

任务分配（pending\_assign → pending）前，系统检查关联 battle\_card 的溯源完整性：

|检查项|规则|缺失影响|
|---|---|---|
|毛利数据|dish\_view.cost 非空 且 有销量|缺失 → blocked|
|销量数据|dish\_sales\_view 有最近周期数据|缺失 → blocked|
|客群基线|store\_baseline 已完成（客群匹配类建议）|缺失 → blocked|
|规则引用|battle\_card.rule\_refs 非空|空 → blocked（建议可信度最高 low，不可执行）|
|关键判断溯源|battle\_card.trace\_ids 非空|空 → blocked|
|delivery\_level|battle\_card.delivery\_level = formal|preview/blocked → 禁止分配|

## 4\.2 blocked 判定与展示

- 任一检查项未过 → 禁止分配，返回 `GATE_BLOCKED` \+ `reasons` 清单。
- 前端在分配按钮处展示 blocked 原因与补齐路径（如"请先同步销量数据"）。
- `battle_card.delivery_level` 为 preview 时，卡片标记"预览版，不可生成任务"。

## 4\.3 例外：老板拍板覆盖

- 老板明确拍板"执行"的建议，即使溯源不完整，可由运营手动覆盖 Gate（需填写 override 原因，记审计日志）。
- 覆盖后 task 标记 `gate_overridden=true`，报告导出时标注"老板拍板覆盖 Gate"。

# 5\. 关闭 Gate（任务关闭前）

## 5\.1 检查项

任务进入 completed/cannot\_execute 后，运营提交复核关闭前检查：

|检查项|规则|缺失影响|
|---|---|---|
|执行回传|task\_execution 存在至少 1 条|无 → blocked|
|执行图片|选"已执行"时 execution\_evidence 非空|空 → blocked|
|执行后指标|baseline\_value / result\_value 至少填一个，或有缺数据说明|全空且无说明 → blocked|

## 5\.2 cannot\_execute 的关闭

- cannot\_execute 任务关闭 Gate：需有 not\_executed\_reason，图片非必须。
- 复核 effect\_label 标 `uncertain` 或 `negative`。

## 5\.3 关闭 blocked 处理

- 未过 Gate → 不可关闭，`task.close_blocked_reasons` 记录原因清单。
- 前端在关闭按钮处展示原因，引导补齐（如"请上传执行图片"）。
- 补齐后重新校验，通过即可关闭。

# 6\. 店长 iLink 身份绑定与对话式回传

> V2.0 这一期不做小程序，店长通过微信 iLink Bot 接入。身份由 `ilink_binding` 表绑定（weixin\_user\_id ↔ store\_staff），任务通知通过 iLink 推送，执行回传为对话式（发图片 \+ 文字 → 智能体解析）。

## 6\.1 店长账号（store\_staff）

- 店长账号由运营在 Web 后台创建（姓名/手机号），关联到一个门店（store\_id）。
- 店长不直接登录本系统；通过微信加 AI 助手号为好友，回复手机号完成 iLink 身份绑定。
- `ilink_binding` 记录绑定关系：`weixin_user_id`（iLink userId）↔ `role=staff` ↔ `ref_id=store_staff.id`。
- 账号状态：`active`（可用）/ `disabled`（停用，消息不再被识别为有效回传）。

## 6\.2 身份绑定流程

```Plaintext
1. 运营在门店详情创建店长账号（姓名/手机号）→ store_staff 记录
2. 店长在微信加品牌 AI 助手号为好友，发任意消息
3. iLink 网关收到消息 → 查 ilink_binding 发现未绑定 → 回复"请回复手机号完成绑定"
4. 店长回复手机号 → 系统按手机号匹配 store_staff
5. 匹配成功 → 写入 ilink_binding（weixin_user_id, role=staff, ref_id=store_staff.id）
   回复"绑定成功！我是 XX 店 AI 助手，有任务会通知你"
6. 匹配失败 → 回复"未找到账号，请联系运营开通"
```

## 6\.3 任务绑定店长

- 任务分配时，运营选择本门店店长账号，`task.assignee_staff_id` 绑定。
- 分配后 iLink Bot 推送任务通知到该店长微信（按 weixin\_user\_id 查 ilink\_binding）。
- 撤销分配 → 解除 `assignee_staff_id` 绑定，后续推送不发送给原店长。
- 重新分配 → 重新绑定店长账号，iLink 推送新通知。

## 6．4 任务归属校验（每次 iLink 回传）

```Plaintext
1. iLink 网关收到店长微信消息 → 回调智能体网关（/ilink/webhook）
2. 按 weixin_user_id 查 ilink_binding：
   - 未绑定 → 拒绝，引导绑定
   - role != staff → 拒绝（可能是老板账号）
   - ref_id 对应的 store_staff.status = disabled → 拒绝
3. 按 store_staff.store_id + assignee_staff_id 查询待执行/执行中任务列表 → 返回匹配的任务
4. 智能体解析店长自然语言 + 图片为结构化回传 → 写入 task_execution
5. 通过 → 记 ilink_access_log，回复店长"已收到回传，已记录"
```

## 6．5 智能体自然语言回传解析

店长在微信回复自然语言 \+ 图片，智能体（LLM + 规则）解析为结构化回传：

| 解析项 | 提取方式 | 示例输入 → 输出 |
|---|---|---|
| 执行状态 | 关键词匹配 | "已执行"/"执行了"/"换了" → executed；"没做"/"做不了" → not\_executed/cannot\_execute |
| 执行人 | ilink\_binding 查 store\_staff.name | 自动填入（店长可文字覆盖，如"小王代执行"） |
| 执行日期 | 消息时间戳 | 取 iLink 消息 received\_at 的日期 |
| 实际动作 | LLM 摘要提取 | "把麻婆豆腐换成了蟹粉豆腐，菜谱已更新" → "替换麻婆豆腐为蟹粉豆腐" |
| 未执行原因 | LLM 提取 | "供应商蟹粉断货了" → "供应商蟹粉断货" |
| 执行后指标 | 正则 + LLM | "这周31份毛利55%" → result\_value=31, 毛利率变化描述 |
| 图片 | iLink CDN 下载 | 微信消息中的图片自动下载存储，关联到 task\_execution |

- 解析由 LLM 完成，输出必须经 schema 校验（含 execution\_status / actual\_action / result\_value 等）。
- 解析失败 → `parsed=false`，标记"待人工复核"，运营在复核页手动补录。
- 同一条微信消息可能包含多个任务的回传（店长一次回复多个任务），智能体按关键词 + 菜品名去重匹配。

## 6．6 店长账号管理

- **创建**：运营在门店详情创建店长账号（姓名/手机号），店长在微信加号完成绑定。
- **停用**：运营停用账号（status=disabled），该店长后续消息不再被识别为有效回传；已分配任务可重新分配。
- **换绑**：店长换微信号，运营删除旧 `ilink_binding`，店长用新号重新加好友 + 回复手机号绑定。
- 账号变更、绑定/解绑均记操作审计。

## 7.1 记录时机

|动作|ilink\_access\_log.action|
|---|---|
|iLink 推送任务通知|`push_notify`|
|收到店长微信回复|`receive_reply`|
|智能体解析回传|`parse_submit`|
|iLink 图片下载存储|`upload_image`|
|身份绑定 |`bind_identity`|

## 7.2 记录字段

- `task_id` / `weixin_user_id` / `role`(boss/staff) / `action` / `success` / `ip` / `error_reason` / `raw_preview`(消息摘要，脱敏) / `accessed_at`

## 7.3 日志用途

- 审计：谁何时收到推送/回复/提交了回传。
- 安全：异常访问检测（如未绑定用户反复尝试）。
- 运营：查看店长是否已读任务通知（push\_notify 后是否有 receive\_reply），催办依据。

# 8\. 执行回传版本管理

## 8\.1 版本快照

- 每次提交创建新 `task_execution` 记录，`version_no` 递增。
- 不覆盖历史记录，所有版本保留。
- 新提交后，旧版本 `is_latest=false`，新版本 `is_latest=true`。

## 8\.2 执行状态取值

- 执行状态以**最新版本**（is\_latest=true）为准。
- 历史版本的状态仅作记录，不影响任务状态机。
- 例：店长先提交"已执行"，后补充证据再提交"已执行"，两次都保留；若改为"无法执行"，最新版本驱动任务流转。

## 8\.3 版本查看

- 任务详情页展示版本时间线，可查看每个版本的回传内容与图片。
- 复核基于最新版本，但可参考历史版本。

## 8\.4 图片与版本的关系

- 图片归属具体 `task_execution_id`（version）。
- 补充上传图片走 `upload_evidence` 接口，挂到最新版本或指定版本。
- 历史版本的图片不可删除，仅可追加。

# 9\. 图片上传安全

## 9\.1 限制规则

|限制项|规则|
|---|---|
|文件类型|仅 jpg / png（白名单）|
|单文件大小|< 5MB|
|单次数量|最多 5 张|
|真实类型校验|通过文件头 magic bytes 校验，不可仅凭扩展名|
|路径暴露|不返回本地绝对路径，仅返回签名 access\_url|

## 9\.2 magic bytes 校验

```Plaintext
jpg: FF D8 FF
png: 89 50 4E 47 0D 0A 1A 0A
```

- 读取文件前 8 字节与白名单 magic bytes 比对，不符拒绝。
- `execution_evidence.mime_verified=true` 后才可入库。

## 9\.3 存储与访问

- 文件存储到本地文件系统或对象存储，内部路径存 `file_path`。
- 对外返回 `access_url`（签名 URL，带过期时间），不暴露 `file_path`。
- `access_url` 过期后需重新请求获取新签名 URL。

## 9\.4 安全规则汇总

|风险|措施|
|---|---|
|越权访问他人任务|iLink 身份绑定 \+ 任务归属校验（assignee\_staff\_id \+ 门店）|
|未授权访问|店长账号需运营创建，无账号无法完成 iLink 身份绑定；账号可停用|
|恶意文件上传|白名单 \+ magic bytes \+ 大小/数量限制|
|路径遍历|不返回绝对路径，用签名 URL|
|重复覆盖|版本快照不覆盖|
|无审计|全量访问日志（ilink\_access\_log）|

# 10\. 复核流程

## 10\.1 复核步骤

1. 运营在任务中心查看已完成回传的任务列表
2. 进入复核页，查看执行前后指标对比（最新版本）
3. 查看执行图片
4. 标记执行效果：`positive` / `negative` / `uncertain`
5. 若数据缺口，标记 `data_gap`：`no_gap` / `observation_pending` / `data_missing`
6. 提交复核 → 写 review\_result → 触发关闭 Gate → 通过则任务关闭

## 10\.2 指标对比自动计算

|指标|执行前|执行后|变化|
|---|---|---|---|
|销量（周均）|task\_execution.baseline\_value|task\_execution.result\_value|自动算百分比|
|毛利率|从 SaaS 取执行前|从 SaaS 取执行后（观察期满后）|自动算 pp|

- 数据缺口自动检测：当前日期 < execution\_date \+ observation\_period\_days → `observation_pending`。
- 执行效果由运营人工标记，不自动判定。

## 10\.3 复核与回写联动

- 复核 `positive` 的任务 → 可归档为案例卡（填适用场景/可复用结论/不可迁移边界）。
- 复核 `negative` 的任务 → 关联规则卡的 `negative_count` \+ 1，规则状态可能转 `needs_correction`。
- 复核结果联动规则/案例/红线回写模块。

# 11\. 老板拍板与反馈联动

## 11\.1 拍板触发

- battle\_card `boss_decision_needed=true` 且运营点"发给老板拍板" → 智能体网关 → iLink Bot `sendmessage` 推送拍板请求到老板微信。
- 推送内容含建议内容 \+ 回复引导（"请回复：同意 / 不同意 / 你的意见"）。
- 详见《微信 iLink Bot 接入与多端对话详细设计》第 7 节。

## 11\.2 拍板结果处理（老板微信自然语言回复）

老板在微信直接回复，智能体解析为决策：

|老板回复|解析|处理|
|---|---|---|
|"同意"/"可以"/"行"|agree|battle\_card.operator\_status=confirmed，可生成任务|
|"不同意"/"不行"/"否"|disagree|operator\_status=rejected|
|"换小份"/"保留麻婆豆腐改小份"|revision|记为 feedback，运营据此调整建议生成新版本|
|其他文字|revision|记为 feedback(feedback\_type=revision)，运营跟进|

- 解析由智能体（LLM）完成，输出结构化 `{decision, content}`，schema 校验，失败转人工
- 解析结果写 feedback（is\_boss\_decision=true）
- 同步更新 battle\_card 状态
- 回复老板"已收到您的决策：…"

## 11\.3 拍板覆盖

- 老板拍板"同意执行"可覆盖溯源完整性 Gate（见 4\.3）。
- 拍板决策内容记入 feedback.is\_boss\_decision=true / boss\_decision\_content。
- 拍板后任务状态更新为"老板决策\-已执行"（若老板直接定方案）。

# 12\. 异常与边界

|场景|处理|
|---|---|
|店长未打开任务超期|运营可催办（基于 open 日志判断未查看），可撤销分配重新分配|
|店长重复提交相同回传|每次均创建新版本（幂等由前端控制，后端不拦截重复）|
|图片上传中断|支持断点续传或重新上传，已成功上传的图片保留|
|任务分配后 SaaS 数据变化（菜品下架）|任务仍可执行，复核时标注"执行对象已变化"|
|老板拍板超时无响应|iLink 3 天未响应再推送提醒 1 次，7 天未响应标"拍板超时"由运营跟进|
|iLink 推送失败（会话过期/context 失效）|降级为运营电话/微信沟通后代录决策；iLink 重新扫码后恢复|
|老板回复解析失败|转人工，运营跟进确认决策|
|Gate 覆盖滥用|老板拍板覆盖需审计，运营手动覆盖需填 override 原因 \+ 二次确认|

# 15\. 开发注意事项

1. **店长身份绑定**：店长通过 iLink 微信 Bot 接入，身份由 `ilink_binding` 绑定（weixin\_user\_id ↔ store\_staff），无需本系统签发登录态。账号停用后 iLink 消息不再被识别为有效回传。
2. **任务归属强校验**：iLink 回传时必须校验 `task.assignee_staff_id` 对应店长且门店一致，越权拒绝。
3. **智能体解析容错**：LLM 解析自然语言回传失败时标记 `parsed=false`，不阻断回传记录。运营在复核页手动补录结构化字段。
4. **版本号原子递增**：task\_execution.version\_no 需原子递增，并发提交用行锁或唯一约束兜底。
5. **is\_latest 维护**：新版本入库时事务内将旧版本 is\_latest 置 false。
6. **图片先校验后入库**：iLink 消息中的图片下载后 magic bytes 校验通过才落盘，避免恶意文件占空间。
7. **Gate 检查复用**：溯源完整性 Gate 与关闭 Gate 抽象为可复用校验函数，分配/关闭/报告导出三处复用。
8. **老板拍板覆盖审计**：gate\_override 必须留审计日志（操作人/原因/时间/老板决策内容）。
9. **终态保护**：completed/cannot\_execute 不可回退，前端隐藏回退入口，后端拒绝回退请求。
10. **iLink 推送降级**：任务分配/拍板请求通过 iLink Bot 推送到店长/老板微信。iLink 会话过期无法推送时降级为运营电话/微信沟通后代录。不依赖小程序。
