# 智能体对话交互详细设计 V1\.0

> 本文档是《产品需求方案 V2\.0》中"智能体对话"模块的开发前详细设计，定义会话模型、交互流程、Agent 能力编排、Prompt 规范、流式协议、结论呈现、优化建议卡片、溯源交互、消息数据结构、异常降级与接口契约。
>
> 多端说明：智能体对话能力复用于三端——运营 Web 对话工作台（本文主体）、店长微信对话（iLink Bot）、老板微信对话（iLink Bot）。店长和老板微信对话均通过 iLink 网关接入，详见《微信 iLink Bot 接入与多端对话详细设计》。message 表以 `channel` 字段区分来源端（web/ilink）。

# 1\. 设计目标

1. **对话即闭环入口**：运营在对话中完成诊断 / 健康度 / 套餐审核 / 优化建议，无需切换独立页面。
2. **多端统一智能体**：同一智能体后端服务运营 Web、店长微信（iLink Bot）、老板微信（iLink Bot）三端，对话跨端可关联。
3. **可溯源**：每条结论与建议都附带溯源（知识库 / SaaS 接口 / 网络查询 / LLM / 人工），实时展示且持久化。
4. **可控边界**：LLM 只生成结构化候选判断，规则卡 / 红线卡优先级高于 LLM，无溯源不得 high confidence。
5. **可执行**：对话产出的优化建议（作战卡）可一键生成任务，进入执行闭环。
6. **可降级**：LLM 失败 / 超时 / 超预算、SaaS 数据缺失时自动降级，不中断主流程。

# 2\. 会话与上下文模型

## 2\.1 会话（Conversation）

- 一个会话绑定**一个门店**，作为对话的稳定上下文。
- 一个门店可有多个会话（并行分析不同问题），互不干扰。
- 会话内可指定**分析范围**：`all`（全店）/ `combo`（指定套餐）/ `dish`（指定菜品），范围可随对话调整。
- 会话状态：`active`（活跃）/ `archived`（归档）。

## 2\.2 上下文（Context）

每次发送消息时，系统为智能体组装上下文，包含：

|上下文项|来源|说明|
|---|---|---|
|门店配置|本系统|定位标签 / 主力时段 / 客单价|
|客群基线|本系统|年龄段 / 偏好 / 频次 / 场景 / 人均|
|品牌红线|本系统|招牌保护清单 / 不可触碰项 / 毛利红线|
|套餐审核配置|本系统|目标客群 / 老板目标 / 平台团购 / 时段|
|SaaS 数据快照|SaaS 只读聚合|菜品 / 成本 / 销量 / 套餐构成，附 synced\_at|
|规则库摘要|FastGPT|按当前问题检索的相关规则卡 / 案例卡引用|
|会话历史|message 表|最近 N 轮消息（见 2\.3）|
|会话摘要|自动生成|长对话压缩的关键结论，避免上下文爆炸|

## 2\.3 上下文窗口与摘要策略

- **滑动窗口**：默认保留最近 10 轮（user\+assistant）原始消息注入 Prompt。
- **摘要触发**：当历史超过 20 轮时，将较早消息压缩为 `session_summary`（关键结论 / 已生成建议 / 老板拍板项），注入 System Prompt。
- **关键结论不丢**：已生成的优化建议（battle\_card）即使消息滑出窗口，仍通过 `battle_card` 表持久化，可在对话内"我的建议"侧栏随时引用。
- **Token 预算**：上下文 \+ Prompt 总 token 超过模型上限 70% 时，优先压缩 SaaS 数据明细（改为聚合统计值），保留规则引用与红线。

## 2\.4 上下文失效与刷新

- 客群基线 / 品牌红线 / 套餐审核配置变更 → 下条消息生效，系统在回复中提示"配置已更新，结论基于新配置"。
- SaaS 数据 → 工具按需调用（带 Redis 缓存），会话内溯源带 `cached_at`；缓存过期（菜品/套餐 24h、销量 4h）则下次调用自动重新拉取。运营可手动刷新缓存。

# 3\. 对话工作台界面

## 3\.1 布局（三栏）

```Plaintext
┌─────────────┬───────────────────────────────┬──────────────────┐
│  左栏：会话  │        中栏：对话流             │  右栏：建议/溯源   │
│             │                                │                  │
│ [新建对话]   │  系统欢迎 / 快捷指令            │  本次会话建议清单  │
│             │                                │  - 建议卡片(缩略)  │
│ 门店A        │  ┌─ 用户消息 ─────────────┐    │                  │
│  ├会话1(活跃)│  │ 帮我诊断门店A菜单        │    │  选中建议的溯源    │
│  └会话2      │  └─────────────────────────┘    │  - 知识库 R012    │
│             │  ┌─ 助手消息 ─────────────┐    │  - SaaS 销量接口   │
│ 门店B        │  │ [诊断结论卡片]           │    │  - 网络查询...     │
│  └会话1      │  │ [健康度卡片]             │    │                  │
│             │  │ [优化建议卡片×N]         │    │                  │
│             │  │ [溯源标记]               │    │                  │
│             │  └─────────────────────────┘    │                  │
│             │                                │                  │
│             │  ┌─ 输入框 ───────────────┐    │                  │
│             │  │ 快捷指令 / 自然语言      │    │                  │
│             │  └─────────────────────────┘    │                  │
└─────────────┴───────────────────────────────┴──────────────────┘
```

- **左栏**：按门店分组的会话列表，新建对话入口。
- **中栏**：对话流，含用户消息、助手消息（含结论卡片 / 建议卡片 / 溯源标记）、输入框与快捷指令。
- **右栏**：本次会话已生成的优化建议清单（缩略卡），点击展开溯源；选中建议可操作"生成任务"。

|`/诊断`|全店菜品多维诊断|`/诊断`|
|`/诊断 菜品名`|单品诊断|`/诊断 麻婆豆腐`|
|`/健康度`|菜单健康度评估|`/健康度`|

|`/建议`|基于已有结论生成优化建议|`/建议`|
|`/对比 A B`|对比两个方案|`/对比 替换蟹粉豆腐 改小份`|

> 自然语言与快捷指令等价，智能体统一走意图识别。

# 4\. 对话交互流程

## 4\.1 单轮消息处理流程（Agent Loop）

```Plaintext
用户发送消息
   │
   ↓
[1] 意图识别（LLM 轻量分类 / 规则匹配）
   │
   ↓
[2] 上下文组装（门店配置 + 客群基线 + 红线 + SaaS快照 + 会话历史 + 摘要）
   │
   ↓
[3] Agent Loop（ReAct / Function Calling 循环）
   │   ├─ 决策调用工具：get_dishes / get_sales / search_rules / web_search / calculate_health ...
   │   ├─ 执行工具，采集 trace（每次工具调用记一条 message_trace）
   │   ├─ 观察结果，决定下一步
   │   └─ 直到信息充分或达到最大步数（默认 6 步）
   │
   ↓
[4] 结论生成（LLM 结构化输出，强制 schema 校验）
   │   ├─ 诊断结论 / 健康度 / 套餐审核结论
   │   ├─ 优化建议卡片（作战卡）
   │   ├─ 附带 evidence_ids / rule_refs / confidence / missing_data
   │   └─ 无溯源 → confidence 强制降级
   │
   ↓
[5] 溯源完整性 Gate 评定 delivery_level（formal / preview / blocked）
   │
   ↓
[6] 流式输出（SSE）：过程 → 结论卡片 → 建议卡片 → 溯源
   │
   ↓
[7] 持久化：message + structured_payload + trace_ids + delivery_level
   │
   ↓
[8] 若产出 battle_card，写入 battle_card 表，右栏同步
```

## 4\.2 多轮交互模式

|模式|触发|行为|
|---|---|---|
|追问|用户对结论追问|复用已有 trace，补充查询；不重复全量诊断|
|调整建议|用户要求调整|基于原建议修订，生成新版本 battle\_card，旧版标记 superseded|
|对比方案|`/对比 A B`|对两个候选方案做并列分析，输出对比表|
|确认建议|用户确认某建议|建议 operator\_status=confirmed，可生成任务|
|跨会话引用|用户引用历史建议|从 battle\_card 表加载，注入上下文|

## 4\.3 Agent Loop 步数与成本控制

- 最大步数：默认 6 步（可配置），超出强制进入结论生成。
- 每步工具调用前检查当日 LLM 累计费用，超阈值（默认 50 元/天）切 dry-run。
- 同一会话 30 秒内重复相同工具调用走缓存（按 prompt\_hash \+ 参数 hash）。

# 5\. 意图识别

## 5\.1 意图分类

|意图|说明|典型工具链|
|---|---|---|
|`dish_diagnose`|菜品多维诊断|get\_dishes → get\_sales → search\_rules → LLM(客群/场景/出品)|
|`menu_health`|菜单健康度|get\_dishes → get\_sales → calculate\_health → search\_rules|
|`combo_audit`|套餐级审核|get\_combos → get\_combo\_config → calculate\_combo → search\_rules → LLM|
|`generate_suggestion`|生成优化建议|依赖已有结论 → search\_rules/cases → LLM|
|`compare`|方案对比|基于已有建议 → LLM|
|`explain`|解释结论/规则|search\_rules → LLM|
|`chitchat`|闲聊/其他|LLM（不调用业务工具，不产出建议）|

## 5\.2 识别策略

- 快捷指令直接映射意图，不走 LLM。
- 自然语言走轻量 LLM 分类（低 temperature），输出意图 \+ 关键槽位（菜品名 / 套餐名 / 方案）。
- 槽位缺失时反问澄清（如"请指定要审核的套餐"），不臆测。

# 6\. Prompt 编排规范

## 6\.1 System Prompt 结构

```Plaintext
你是精品正餐品牌的菜单决策智能体。你的职责是基于门店数据、客群基线、品牌红线与规则库，
完成菜品诊断、菜单健康度评估、套餐级审核，并产出可执行的优化建议。

【角色边界】
- 你只能生成结构化候选判断，最终结果必须符合输出 schema。
- 规则卡与红线卡优先级高于你的文案判断，不得推翻已命中红线卡的禁止动作。
- 没有证据或规则引用时，只能输出"待确认"，confidence 不得为 high（最高 low）。
- 你的输出必须包含 evidence_ids、rule_refs、confidence、missing_data。

【可用工具】
- get_store_config / get_dishes / get_sales / get_combos / get_combo_config
- search_rules / search_cases / web_search
- calculate_health / calculate_combo

【输出规范】
- 诊断结论按结构化 schema 输出（见 schema 定义）。
- 优化建议以卡片形式输出，每条含类型/描述/预期效果/可信度/是否拍板/溯源。
- 每条结论与建议附溯源标记，引用规则卡用 R-编号，案例用 C-编号，红线用 RL-编号。

【当前上下文】
- 门店：{store_name}（{position_tags}）
- 客群基线：{baseline_summary}
- 品牌红线：{redline_summary}
- 套餐审核配置：{combo_config_summary}
- SaaS 数据时效：{synced_at}
- 会话摘要：{session_summary}
```

## 6\.2 动态注入

- **规则摘要**：每轮根据意图检索 Top 5 规则卡，注入编号 \+ 内容摘要（不注入全文，全文按需回查）。
- **红线摘要**：注入当前门店所有 active 红线（全文，因红线优先级最高）。
- **SaaS 数据**：注入聚合统计（菜品数 / 低毛利占比 / 套餐毛利率等），明细按需通过工具获取，避免上下文爆炸。

## 6\.3 输出 Schema（JSON Schema 片段）

助手消息的 `structured_payload` 必须通过以下 schema 校验，校验失败重试 1 次，仍失败则降级（规则兜底 / dry-run）。

```JSON
{
  "type": "object",
  "required": ["intent", "conclusions", "suggestions", "confidence", "missing_data"],
  "properties": {
    "intent": {"type": "string"},
    "conclusions": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["type", "content", "confidence", "rule_refs", "evidence_ids"],
        "properties": {
          "type": {"enum": ["dish_diagnose", "menu_health", "combo_audit"]},
          "content": {"type": "object"},
          "confidence": {"enum": ["high", "medium", "low", "missing_data"]},
          "rule_refs": {"type": "array", "items": {"type": "string"}},
          "evidence_ids": {"type": "array", "items": {"type": "string"}},
          "missing_data": {"type": "array", "items": {"type": "string"}}
        }
      }
    },
    "suggestions": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["suggestion_type", "suggestion_desc", "expected_effect", "confidence", "boss_decision_needed", "rule_refs"],
        "properties": {
          "suggestion_type": {"enum": ["换菜", "下架", "改规格", "新增", "调价"]},
          "suggestion_desc": {"type": "string"},
          "expected_effect": {"type": "string"},
          "confidence": {"enum": ["high", "medium", "low", "missing_data"]},
          "boss_decision_needed": {"type": "boolean"},
          "rule_refs": {"type": "array", "items": {"type": "string"}},
          "case_refs": {"type": "array", "items": {"type": "string"}},

    "missing_data": {"type": "array", "items": {"type": "string"}}
  }
}
```

# 7\. 工具定义（Function Calling）

所有工具调用记入 `message_trace`，source\_type 对应见 11\.2。

## 7\.1 SaaS 数据工具

|工具|入参|出参|source\_type|
|---|---|---|---|
|`get_store_config`|store\_id|门店配置 \+ 客群基线 \+ 红线|本系统|
|`get_dishes`|store\_id, \[category\]|菜品视图（实时聚合 SaaS 流水，Redis 缓存 24h）|saas\_api|
|`get_sales`|store\_id, item\_id, period|销量序列（实时聚合，缓存 4h）|saas\_api|
|`get_combos`|store\_id|套餐构成 \+ 价格（实时聚合，缓存 24h）|saas\_api|
|`get_combo_config`|store\_id, combo\_id|套餐审核配置|本系统|

> SaaS 工具不落库主数据，缓存优先，miss 则拉 SaaS 流水实时聚合。详见《SaaS 只读接入详细设计》。SaaS 不可用且缓存 miss 时返回 missing\_data，由溯源完整性 Gate 降级。

## 7\.2 知识库工具（FastGPT）

|工具|入参|出参|source\_type|
|---|---|---|---|
|`search_rules`|query, top\_k|规则卡引用列表（编号 \+ 摘要）|knowledge\_base|
|`search_cases`|query, top\_k|案例卡引用列表|knowledge\_base|

> 检索只返回引用，智能体按编号回查结构化资产原文；不得把检索摘要当结论。

## 7\.3 计算工具（本地确定性）

|工具|入参|出参|说明|
|---|---|---|---|
|`calculate_health`|store\_id|总分 \+ 各维度得分 \+ 风险列表|确定性计算，不走 LLM|
|`calculate_combo`|store\_id, combo\_id|套餐毛利率 \+ 套餐级风险|确定性计算|

> 毛利 / 销量 / 健康度 / 套餐毛利率等数值计算必须走本地确定性工具，LLM 不得自行算数。

## 7\.4 外部工具

|工具|入参|出参|source\_type|
|---|---|---|---|
|`web_search`|query|查询结果摘要 \+ 来源 URL|web\_search|

> 网络查询结果只能作为辅助参考，不得作为 high confidence 依据；引用时记 trace。

## 7\.5 工具调用示例（套餐审核）

```Plaintext
intent: combo_audit
Step1: get_combos(store_id) → 套餐列表
Step2: get_combo_config(store_id, combo_id) → 审核配置
Step3: calculate_combo(store_id, combo_id) → 套餐毛利率 + 风险（确定性）
Step4: search_rules("套餐低毛利 客群不匹配") → R012, R023
Step5: LLM(菜品明细 + 客群基线 + R012) → 单菜匹配判断（结构化，含 missing_data）
Step6: 汇总 → 套餐整体结论 + 单菜建议（schema 校验）
```

# 8\. 流式输出协议（SSE）

消息发送接口 `POST /api/v1/conversations/{id}/messages` 返回 `text/event-stream`，事件类型：

|event|data|说明|
|---|---|---|
|`thinking`|`{"step": "调用 get_sales...", "tool": "get_sales"}`|Agent 思考/工具调用过程|
|`trace`|`{"trace_id": "tr_01", "source_type": "knowledge_base", "source_ref": "R012"}`|单条溯源（随工具调用实时推送）|
|`content_delta`|`{"text": "..."}`|自然语言文本增量|
|`conclusion`|`{"type": "combo_audit", "payload": {...}}`|结构化结论卡片|
|`suggestion`|`{"battle_card_id": "bc_01", "payload": {...}}`|优化建议卡片|
|`delivery`|`{"delivery_level": "formal", "blocked_reasons": []}`|溯源完整性 Gate 结果|
|`done`|`{"message_id": "m_01"}`|结束|

**前端渲染顺序**：thinking（可折叠过程区）→ content\_delta（正文）→ conclusion（结论卡片）→ suggestion（建议卡片）→ delivery（Gate 标记）→ done。trace 实时累积到结论/建议的溯源区。

# 9\. 诊断结论呈现规范

## 9\.1 菜品多维诊断

助手消息中以**诊断表格卡片**呈现（非纯文本）：

```Plaintext
【菜品多维诊断 · 门店A】
| 菜名 | 毛利 | 销量 | 品牌角色 | 客群匹配 | 场景匹配 | 出品 | 组合 |
| 麻婆豆腐 | 低毛利↓ | 低销量↓ | 陪衬菜 | 不匹配⚠ | 不匹配⚠ | 中 | 无搭配 |
| 宫保鸡丁 | 高毛利 | 高销量↑ | 招牌菜 | 匹配 | 匹配 | 中 | 核心搭配 |
...
[展开溯源] [导出 CSV]
```

# 10\. 优化建议卡片（作战卡）交互

## 10\.1 卡片结构

每条优化建议以独立卡片嵌入助手消息，字段：

|字段|展示|说明|
|---|---|---|
|建议类型|标签|换菜/下架/改规格/新增/调价|
|建议描述|正文|具体操作|
|预期效果|正文|如"毛利率 28%→55%"|
|可信度|标签|high/medium/low（颜色区分）|
|是否需拍板|标签|需拍板时高亮|
|溯源|折叠区|rule\_refs / case\_refs / trace|
|delivery\_level|标记|formal / preview|

## 10\.2 卡片操作

|操作|行为|
|---|---|
|**确认**|operator\_status=confirmed，卡片变绿，"生成任务"按钮可用|
|**驳回**|operator\_status=rejected，记录原因|
|**要求调整**|进入多轮调整，生成新版本，旧版 superseded|
|**生成任务**|仅 confirmed 可用 → 创建 task，跳转任务中心或弹窗编辑|
|**发给老板拍板**|需拍板项 → iLink 推送拍板请求到老板微信，老板自然语言回复|
|**批量确认**|右栏多选建议 → 批量 confirmed → 批量生成任务|

## 10\.3 版本管理

- 调整建议生成新 `battle_card` 记录，`superseded_by` 指向新版；旧版保留可追溯。
- 同一对象（套餐/菜品）的最新版建议为有效建议。

# 11\. 溯源交互

## 11\.1 展示

- 结论 / 建议末尾显示溯源标记（如 `📎 R012 · SaaS销量 · 网络查询×1`）。
- 点击展开溯源详情面板（右栏或浮层）：

```Plaintext
溯源详情 · "麻婆豆腐客群不匹配"
- 知识库引用：R012 低毛利+客群不匹配→替换  [查看规则卡]
- SaaS 接口：get_sales(门店A, 麻婆豆腐, 近4周)  销量156  [查看数据]
- LLM 推理：客群匹配判断 confidence=medium  missing_data=[客群调研]
- 人工标注：招牌菜保护清单（宫保鸡丁）
```

## 11\.2 溯源类型与采集

|source\_type|采集时机|记录内容|
|---|---|---|
|`knowledge_base`|search\_rules/cases 调用|source\_ref(R/C/RL编号) \+ snippet \+ query|
|`saas_api`|SaaS 数据工具调用|工具名 \+ 参数 \+ 返回摘要|
|`web_search`|web\_search 调用|query \+ 结果摘要 \+ URL|
|`llm`|LLM 生成判断|input\_summary \+ output \+ confidence \+ missing\_data|
|`manual`|引用人工标注|标注项 \+ 标注人|

## 11\.3 溯源完整性 Gate（详见 PRD V2\.0）

- formal：数据完整 \+ 规则引用齐全 \+ 关键判断有溯源
- preview：任一缺失，标记预览版，列 blocked 原因
- blocked：关键数据（毛利/销量）缺失，不可生成任务

# 12\. LLM 判断边界落地

|边界|落地实现|
|---|---|
|结构化输出|structured\_payload 经 6\.3 schema 校验，失败重试 1 次，再失败降级|
|必含字段|schema required 强制 evidence\_ids / rule\_refs / confidence / missing\_data|
|可信度约束|后处理：若 rule\_refs 为空 → confidence 强制 ≤ low；若 missing\_data 含毛利/销量 → missing\_data|
|规则/红线优先|红线全文注入 System Prompt；命中红线禁止动作时，LLM 输出若冲突则采信红线|
|失败回退|LLM 失败/超时/超预算 → 规则兜底（确定性规则）→ dry-run，记 fallback\_used|
|调用留痕|每次 LLM 调用写 llm\_call\_log（input\_summary/output/schema\_valid/cost/fallback\_used）|

# 14\. 异常与降级

|异常|降级策略|用户感知|
|---|---|---|
|LLM 超时/失败|规则兜底（确定性规则给出结论）→ dry-run|提示"AI 暂不可用，已基于规则给出结论（可信度 medium）"|
|LLM 超预算|当日切 dry-run|提示"已达当日 AI 预算，使用模拟模式"|
|SaaS 接口失败|用最近缓存，标记 synced\_at 时效|溯源标记"数据时效 X 小时"，delivery=preview|
|毛利/销量缺失|维度标"数据缺失"，delivery=preview/blocked|blocked 原因列"销量数据未同步"|
|schema 校验失败|重试 1 次 → 降级|提示"结论生成异常，已降级"|
|Agent 步数超限|强制进入结论生成|提示"已基于当前信息给出结论，部分维度待确认"|

## 16\.4 错误码

|code|说明|
|---|---|
|`GATE_BLOCKED`|溯源完整性 Gate 未通过，禁止生成任务|
|`LLM_DEGRADED`|LLM 降级（兜底/dry-run），结果可信度可能降低|
|`SAAS_STALE`|SaaS 数据过期，结论为 preview|
|`SCHEMA_INVALID`|LLM 输出 schema 校验失败已降级|
|`CONTEXT_INCOMPLETE`|门店配置/客群基线未完成，无法诊断|

# 17\. 开发注意事项

1. **数值计算不走 LLM**：毛利 / 销量排名 / 健康度 / 套餐毛利率全部本地确定性计算，LLM 只做匹配/判断类推理。
2. **溯源必须可回查**：rule\_refs 必须能回查到 rule\_card 原文，不得只存摘要。
3. **流式与持久化分离**：SSE 实时推送的内容与最终持久化的 message 可能不完全一致（如 thinking 不持久化），done 事件后才落库。
4. **幂等**：同一消息重复发送（网络重试）应幂等，避免重复创建 battle\_card。
5. **审计**：所有 LLM 调用、工具调用、Gate 评定均留痕，便于复盘"为什么给出这个建议"。
